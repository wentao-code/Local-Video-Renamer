# 单一补全执行表设计

## 目标

十张来源独立的待补全表只保存尚未执行或可恢复重试的数据。系统领取一个批次时，将记录在同一 SQLite 事务中物理搬入唯一的 `enrichment_running_items` 表。执行成功或形成业务终态后删除执行记录；网络、连接或程序异常时，将记录事务性退回各自来源表。

## 数据结构

`enrichment_running_items` 保存原待补全行的全部业务字段，并额外保存：

- `origin_table`：固定白名单中的原待补全表名。
- `plan_id`、`sequence_index`：保持原计划身份不变。
- `task_kind`：视频、番号、演员或演员资料。
- `attempt_count`、`claimed_at`、`last_error`：保留恢复和诊断信息。

`enrichment_batch_plans` 增加持久化完成计数。计划总量由“已完成计数 + 来源表剩余行 + 执行表当前行”构成，因此成功删除执行记录后进度不会丢失。

## 数据流

1. 创建计划时，候选记录只写入对应来源待补全表。
2. 领取时使用 `BEGIN IMMEDIATE`。执行表非空且属于其他任务时拒绝领取；否则将限定数量记录插入执行表并从来源表删除，然后提交。
3. 补全服务只使用执行表返回的本批数据。
4. 成功、无搜索结果、缺少演员等业务终态从执行表删除并增加完成计数，不回原表。
5. 单项基础设施失败回原表并保留失败状态及错误；批次异常、网络断开或进程恢复则将仍在执行表中的行退回原表为待处理状态。
6. 启动恢复会清空执行表：逐行按 `origin_table` 退回，再将运行中的计划改为暂停。

## 一致性与安全

- 搬入、成功确认和失败退回都各自在单个写事务中完成。
- `origin_table` 必须属于十张来源表或历史兼容表白名单，禁止任意 SQL 表名。
- 同一时间执行表只能承载一个计划；同计划恢复读取允许返回既有执行行。
- 历史计划可从旧混合表领取，但进入统一执行表后遵循同样规则。

## 测试

- 领取后原表减少、执行表增加，且字段保持不变。
- 执行表非空时其他计划无法领取。
- 成功后执行表记录消失且不回原表，计划完成数增加。
- 网络/异常退回正确原表且执行表清空。
- 多来源恢复分别返回各自原表。
- 完整补全计划和现有 GUI、来源顺序测试全部通过。

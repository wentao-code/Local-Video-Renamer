# 排除网页影片缓存表设计

## 目标

将番号黑名单、演员黑名单和视频筛选规则排除的网页影片记录从活动网页影片表中分离，排除记录只作为“已抓取”去重依据，不再进入界面、候选计算或任何补全任务。

## 范围与边界

- 保留 `processed_videos` 原样作为本地视频主表，继续承担本地文件去重、扫描、重命名和本地补全状态。
- 保留 `code_prefix_enrichments`、`actor_enrichments` 等实体级来源状态；它们用于记录番号/演员来源是否已经抓取，不迁移到排除表。
- 新增 `excluded_code_prefix_movies`，对应活动表 `code_prefix_movies`。
- 新增 `excluded_actor_movies`，对应活动表 `actor_movies`。
- 排除表保存完整网页影片信息、来源字段、最近抓取时间和排除原因，但不保存或继续更新补全任务状态。

## 排除规则

1. 番号黑名单优先：`hidden_code_prefixes` 命中的番号网页记录进入 `excluded_code_prefix_movies`。
2. 演员黑名单优先：`hidden_actors` 命中的演员网页记录进入 `excluded_actor_movies`。
3. 视频筛选规则：活动网页影片匹配当前筛选器时进入对应排除表。
4. 同一条记录可因多种原因排除，`exclude_reason` 保存稳定的原因集合文本。

## 数据流

抓取结果进入 `replace_code_prefix_movies` 或 `replace_actor_movies` 时，先标准化并合并已有本地/JAVTXT状态，再在同一事务中分流：符合排除规则的记录写入排除表，其他记录写入活动表。排除表记录不会调用补全服务，也不会被活动表的全量替换删除。

所有“是否已抓取”的判断同时查询活动表和排除表，并提供批量查询接口，禁止逐条查询形成 N+1。

## 历史迁移

数据库初始化时只创建表，不执行大批量迁移。提供一次性的后台迁移方法，分批扫描活动表，先插入排除表并校验，再删除活动表记录。迁移支持断点重复执行，完成后返回每张表的移动数量、跳过数量和错误信息。

## 恢复策略

排除记录默认不自动恢复，即使之后删除黑名单或修改筛选规则，也不会重新进入补全。提供显式恢复接口时，恢复前重新判断当前排除规则；只有不再命中任何排除规则的记录才可以回到活动表。

## 一致性要求

- 活动表和排除表使用相同的业务主键：番号为 `(prefix, code)`，演员为 `(actor_name, code)`。
- 分流、移动和恢复操作使用单事务，避免“主表删除成功但排除表写入失败”。
- 排除表记录不出现在活动列表、数据中心、候选库、补全计划候选和补全执行结果中。
- 活动表和排除表都参与已抓取去重查询。

## 验证

- 新抓取记录按规则正确分流。
- 黑名单批量移动可重复执行且不会重复插入。
- 活动表替换不会删除排除表记录。
- 排除记录不会出现在候选和补全计划中。
- 主表与排除表联合去重可阻止重复抓取。
- 迁移失败时事务回滚，完整回归测试通过。
